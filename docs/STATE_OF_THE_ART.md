# KRN State Of The Art

## Status

This is the active project doctrine for the clean KRN direction. It is derived
from the raw onboarding source in `docs/materials/` and should stay smaller
than the raw source.

## Product Promise

`krn init` should eventually turn any repo into an agent-ready,
source-grounded, memory-aware, eval-driven, reviewable Codex CLI working
environment.

KRN wins only if daily Codex work becomes:

- more continuous after compaction;
- less chaotic;
- more source-grounded;
- less costly to review;
- less vulnerable to context rot;
- less vulnerable to stale memory;
- more repeatable without ritual.

## Failure To Avoid

The previous KRN direction failed when it became an artifact factory:

- goals as product brain;
- file-backed pseudo-memory;
- source hoarding;
- dashboard and benchmark surfaces before workflow proof;
- eval theater;
- broad agent taxonomy before output contracts.

The replacement path is:

```text
kernel contract
  -> pattern selection
  -> engineering skills
  -> typed primitives
  -> dry-run init
  -> context packet
  -> Codex run
  -> review capture
  -> Memory/Source/Eval candidates
  -> dogfood
  -> promotion
```

## Operating Brain

KRN is one operating brain around Codex, not many independent brains.

Brain functions:

- problem framing;
- context selection;
- source grounding;
- memory application;
- execution contract;
- policy control;
- review intelligence;
- feedback distillation;
- pattern selection;
- handoff compression.

Subagents, skills, MCP, hooks, evals, and dashboard are organs. They are not
the brain.

## Initial Adopted Patterns

- Small Kernel Contract.
- Raw Material Quarantine.
- Context Supply Chain.
- Memory Is Store-Backed, Files Are Export.
- Source-To-Decision Graph.
- Skills As Engineering Disciplines.
- Built-In Skill Creation Pipeline.
- Progressive Disclosure Skill Design.
- Read / Propose / Write Boundary.
- Sandbox / Approval / Permission Boundary.
- Hook Trust And Deterministic Gates.
- MCP As Typed Tool Boundary, Not Memory.
- Rules As Experimental Command Policy.
- Review Burden + Diff Risk As Product Metrics.
- Trace To Feedback To Eval Candidate To Handoff.
- Review / Repair / Validate Loop.
- Evidence-Checked Continuous Goal.
- TypeScript Boundary Discipline.

## Later Only

- plugin packaging;
- macro-evals over real trace populations;
- Codex GitHub Action or `codex exec` automation;
- KRN MCP server;
- multi-project registry;
- dashboard over typed objects.
